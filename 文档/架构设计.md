# 系统架构设计

本文档详解项目的三层架构设计。

---

## 架构概览

项目通过三层分层架构集成 LLM 与数学优化，形成 LLM 引导的优化循环。

```
User Needs
    ↓
┌───────────────────────────────────────┐
│ Layer 1: LLM 作为元目标设计器          │
│ 动态生成 Python 目标函数代码           │
│ (llmAPI/, prompt/)                    │
└───────────────────────────────────────┘
    ↓
┌───────────────────────────────────────┐
│ Layer 2: Harmony Search 作为提示进化器 │
│ 管理种群，进化优化目标                  │
│ (heuristics/)                         │
└───────────────────────────────────────┘
    ↓
┌───────────────────────────────────────┐
│ Layer 3: 优化器作为约束执行器          │
│ 用数学严谨性求解操作决策               │
│ (model/, simulator/)                  │
└───────────────────────────────────────┘
    ↓
Best Solution
```

---

## Layer 1: LLM 作为元目标设计器

### 位置
- **Main**: `llmAPI/llmInterface.py`, `llmAPI/llmInterface_huggingface.py`
- **Prompts**: `prompt/modPrompt.py`

### 角色

LLM 接收优化场景描述和约束条件，动态生成 Python 目标函数代码。

**输入**：
- 场景描述（任务、资源、约束）
- 优化目标指导
- Gurobi 建模规则

**输出**：
- 可执行 Python 代码
- 目标函数定义（替换 Layer 3 中的标准目标）

### 关键特性

- **动态代码生成** — 每次进化迭代生成新的目标函数
- **Gurobi 规则库** — 15 条约束编写指导
- **多格式响应解析** — 支持 `<think>` 标签、JSON、纯文本
- **Timeout 控制** — 120s 超时防止卡机
- **模型灵活性** — 支持 qwen3.5-plus, deepseek-chat, gpt-4o 等

## Layer 2: Harmony Search 作为提示进化器

### 位置
- **Framework**: `heuristics/hsFrame.py`
- **Population**: `heuristics/hsPopulation.py`
- **Individuals**: `heuristics/hsIndividualMultiCall.py` (MoD), `heuristics/hsIndividualEdgeUav.py` (Edge UAV)
- **Selection**: `heuristics/hsSorting.py`

### 角色

Harmony Search 是种群进化算法，管理 LLM 生成的目标函数种群。

**进化循环**：
```
Initialize population (popSize=5)
  ↓
For each generation (iteration=10):
  ├─ Evaluate each individual (HS 适应度)
  ├─ Sort by fitness
  └─ Generate new harmony (3 种变异策略)

Return best solution
```

### 四种变异策略

| 策略 | 说明 |
|------|------|
| **way1** | 随机调整权重系数 |
| **way2** | 修改约束松弛程度 |
| **way3** | 重新采样目标函数参数 |
| **way4** | 组合变异（推荐 Edge UAV） |

### 个体评估流程

每个个体的评估包括：

```
1. LLM 生成目标函数代码
2. 运行 Level 1 优化（BLP with LLM objective）
3. 运行 Level 2 优化（Sequencing）
4. 仿真评估（真实系统性能）
5. 返回适应度分数
```

## Layer 3: 优化器作为约束执行器

### Level 1: 任务分配（Binary Linear Program）

**位置**: `model/two_level/AssignmentModel.py` 或 `edge_uav/model/offloading.py`

**角色**: 在 LLM 目标下，求解任务到资源的最优分配

**关键特性**：
- 二进制决策变量（$x_{ij}$ = 是否分配任务 i 到资源 j）
- LLM 目标通过 `exec()` 动态注入
- Gurobi 求解器（商业优化库）
- 约束：唯一分配 + 可选承载上限
- 当前实现不再把超时写成独立硬约束；超时通过目标函数中的归一化时延项软惩罚
- 当前输出格式为 `{t: {"local": [...], "offload": {j: [...]}}}`，不含 `drop`

### Level 2a: 资源分配（凸优化）

**位置**: `edge_uav/model/resource_alloc.py`

**角色**: 在固定卸载决策下，优化边缘 CPU 频率分配

**特性**：
- 当前实现将本地频率固定为 `task.f_local`，不把 `f_local` 作为显式优化变量
- 连续凸优化变量为边缘频率 $f_{j,i}^t$
- 求解方法为闭式 KKT + 容量受限时的对偶二分法

### Level 2b: 轨迹优化（SCA）

**位置**: `edge_uav/model/trajectory_opt.py`

**角色**: 在固定资源下，用连续凸近似（SCA）优化 UAV 轨迹

**特性**：
- 非凸问题 → SCA 线性化
- 迭代求解（通常 10 次收敛）
- 约束：通信延迟、能耗、安全分离
- **注意**：Phase⑥ Step3 正修复 DCP 非凸性问题

### 仿真（System Evaluator）

**位置**: `simulator/SimClass.py` 或 `edge_uav/model/evaluator.py`

**角色**: 评估优化解的真实系统性能

**计算内容**：
- 任务完成延迟
- 能耗消耗
- 系统成本函数

**输出**：
- 适应度分数（用于 HS 排序）
- 可行性判断

## 执行流

### MoD 系统（原始）

```
testAll.py
  ├─ 加载 config/setting.cfg
  ├─ 生成场景（scenarioGenerator.py）
  ├─ 运行 Harmony Search（hsFrame.py）
  │   └─ For each generation:
  │       └─ For each individual:
  │           ├─ LLM 生成目标（Layer 1）
  │           ├─ Assignment + Sequencing（Layer 3）
  │           └─ 评估适应度（SimClass）
  └─ 输出最优解（discussion/）
```

### Edge UAV 系统（新）

```
testEdgeUav.py
  ├─ 加载 config/setting.cfg + config/config.py
  ├─ 生成 Edge UAV 场景（edge_uav/scenario_generator.py）
  ├─ 运行 Harmony Search（hsFrame.py）
  │   └─ For each generation:
  │       └─ For each individual (hsIndividualEdgeUav.py):
  │           ├─ LLM 生成目标（Layer 1）
  │           ├─ Offloading（Level 1 BLP）
  │           ├─ Resource Allocation（Level 2a）
  │           ├─ Trajectory Optimization（Level 2b）
  │           └─ 评估（evaluator.py）
  └─ 输出最优解（discussion/）
```

## Edge UAV 系统特性

### 问题定义

**目标**: 最小化任务延迟、边缘计算能耗、UAV 飞行能耗的加权和

**优化变量**：
- $x_{ij}^t$ — 任务卸载决策（二进制）
- $f_{ji}^t$ — 边缘 CPU 频率分配（连续）
- $q_j^t$ — UAV 轨迹位置（连续，2D）

**当前实现说明**：
- 本地频率由场景中的 `task.f_local` 给定，不在 `resource_alloc.py` 中单独优化

### 二层分解

**原问题**: 混合整数非凸 → 难以求解

**分解策略**（Block Coordinate Descent）：
```
Fix (q, f_edge) → Optimize offloading (x)
Fix (x, q) → Optimize edge frequency (f_edge)
Fix (x, f_edge) → Optimize trajectory (q)
...iterate until convergence
```

### 数据类型

| 类 | 位置 | 说明 |
|------|------|------|
| `ComputeTask` | `edge_uav/data.py` | 计算任务定义 |
| `UAV` | `edge_uav/data.py` | 无人机定义 |
| `EdgeUavScenario` | `edge_uav/data.py` | 场景容器 |

### 配置参数

所有 42 个 Edge UAV 参数集中在 `config/config.py`，分为 10 个部分：
- 时间/实体配置
- 能耗参数
- 通信参数
- 安全参数
- 等等

详见 `config/config.py` 及项目 CLAUDE.md

## 关键设计决策

| 决策 | 理由 |
|------|------|
| **BCD 分解** | 混合整数问题 → 可分解求解 |
| **外部评估器** | 不同目标函数的适应度不可比 |
| **Way4 策略** | Edge UAV 特化的变异方式 |
| **SOCP 改写** | 解决 DCP 约束非凸性 |
| **CVXPY** | 凸优化的标准框架 |
