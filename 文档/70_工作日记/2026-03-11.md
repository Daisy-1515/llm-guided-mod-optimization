### 上午：探索原项目 prompt 机制

重新精读了原项目的完整 prompt 体系：

- `basicPrompt.py`：五大组件（scenario / static_map_info / init_level1model / obj_format / cons_restriction）
- `modPrompt.py`：三种演化策略（way1 全新生成 / way2 在最优个体基础上改进 / way3 彻底重构）
- `AssignmentModel.py`：通过 `exec()` 将 LLM 生成的函数字符串注入 `setupObj()`
- `hsIndividual.py`：将 taxi/passenger 状态转为字符串，拼入动态 prompt

关键理解：LLM 不是生成整个解，而是生成一个 **Gurobi 目标函数代码片段**（`dynamic_obj_func`），然后由 Gurobi 精确求解该目标下的最优 $x$。

### 下午：三方头脑风暴

向 Gemini 和 Codex 并行发起头脑风暴，核心问题是 Edge UAV 场景的独特挑战。

**Gemini 的关键贡献：**
1. 时隙耦合缺失：LLM 可能生成"贪婪型"目标，忽略跨时隙资源预留
2. 量级差异：时延（ms级）vs 能耗（J级）需要在 prompt 里明确说明
3. Deadline 必须在目标函数里显式惩罚（而非隐含约束）
4. 建议新增 **way4（资源感知策略）**：引导 LLM 感知 UAV 剩余电量、实现负载均衡
5. 提供了 `prompt_scenario` 草稿文本

**Codex 的关键贡献：**
1. 三维变量 `x_offload[i,j,t]` 索引顺序错乱是 LLM 的高频错误，需要在 `cons_restriction` 里明确
2. 活跃任务 gating：所有求和必须用 `task[i].active[t]` 过滤
3. 提供了 `default_dynamic_obj_func` 的完整 Gurobi 代码原型（diff patch）
4. 给出了 9 条新增 `cons_restriction` 规则

**我对原型的审查（关键问题）：**
- 两方原型都错误地引用了父类 `basicPrompts` 的属性（`llmHasMemory`、`prompt_scenario` 等），而 `EdgeUavPrompts` 是全新基类，不继承原项目。需要重写。
- Gemini 原型要求 `dynamic_obj_func` "返回 obj 表达式"，与原项目实际调用方式不符——应是 `self.model.setObjective(...)` 直接调用。
