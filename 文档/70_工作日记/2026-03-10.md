# 2026年3月10日 周二

## 项目：边缘无人机计算卸载系统建模

---

今天是项目的第二天。昨天把数学模型搭好了，今天的主题是**解耦**——把那个庞大的 MINLP 问题拆成可以求解的子问题，然后和原论文做了一次彻底的对比审查。

### 下午：两层解耦方案

下午的核心工作是写 `公式20_两层解耦.md`。公式 (20) 是整个系统的目标函数，同时包含离散的卸载决策、连续的资源分配和 UAV 轨迹——直接联合求解不可行，必须拆开。

拆法参考了 UAV-MEC 领域的标准做法：

- **Level 1**：固定轨迹和资源，只求解卸载决策 $x$。离散变量固定后，所有时延和能耗都变成了已知常数，目标函数关于 $x$ 退化为线性——变成了一个二元线性规划（BLP），可以直接扔给 Gurobi。

- **Level 2**：固定卸载决策，求解轨迹 $\mathbf{q}$ 和资源 $f$。这一层虽然去掉了离散变量，但仍然是非凸的（信道模型里的 $1/\log(1+\text{SNR})$ 和推进功率公式都是非凸的）。所以内部又拆成两个子块：Level 2a 求资源分配（凸优化），Level 2b 求轨迹（SCA 逐次凸近似）。

两层之间通过 BCD（块坐标下降）交替迭代，直到系统成本收敛。

写到 LLM 注入点的时候特别花了些心思。原论文里 LLM 生成的是 Level 1 的代理目标函数——替换掉标准目标，让 Gurobi 按照 LLM 的"直觉"来做分配。我们保持了同样的设计，但需要想清楚 LLM 能看到哪些信息、代理目标的自由度有多大。

### 傍晚：和原论文的深度对比

写完解耦文档后，回头和原论文做了一次系统性对比。这一步用了 Claude + Gemini + Codex 三方协作的头脑风暴模式，从六个维度逐一审查。

发现了几个关键差异：

**1. 我们需要 BCD 迭代，原论文不需要。** 原论文的 ride-hailing 场景里，分配完乘客之后路径规划不会反过来影响分配的可行性——信息是单向流动的。但我们的场景不一样：卸载决策依赖 UAV 位置（通过信道增益），UAV 轨迹又依赖负载分布（由卸载决策决定），是双向耦合。BCD 迭代就是为了在这种强耦合中找到平衡。

**2. 原论文 Level 2 是 MILP，我们是非凸连续优化。** 本质原因是物理模型不同——出租车走的是固定路网（距离矩阵已知），无人机飞的是连续空间（位置影响电磁传播和空气动力学），没法线性化。

**3. 收敛性声明有问题。** 这是 Codex 最先指出来的。原文档 §7 说"BCD 每步非递增所以收敛"，但 Level 1 用的是 LLM 代理目标而非真实目标——代理目标下降不等于真实目标下降，经典 BCD 的单调性保证不成立。

Gemini 给了一个很漂亮的修正思路：把代理目标写成"真实目标 + LLM 修正项"的正则化形式，限制修正项的幅度，再加一个外层的成本回滚机制。这样既保留了 LLM 的灵活性，又不至于跑飞。

最后把 §7 重写了，拆成三小节，分别讨论标准目标下的收敛性、LLM 代理目标下的修正机制、以及方法的整体定位。

### 晚上：整理提交

把对比分析的完整报告整理成了 `文档/解耦对比分析报告.md`（经过 Codex 两轮评审，7.9/10 通过），连同解耦文档和之前的变量修正一起提交了。

### 小结

今天最大的收获是**从"写公式"到"审公式"的转变**。写解耦方案的时候觉得一切顺理成章，但和原论文一对比就发现了不少需要重新审视的地方——尤其是收敛性那个坑，如果不做对比很可能一直藏在那里。三个 AI Agent 在这种对比审查中确实各有所长：Gemini 擅长提出创意方案（正则化形式），Codex 擅长挑毛病（条件化表述、计算估算），Claude 负责综合判断和最终决策。
