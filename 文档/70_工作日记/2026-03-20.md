## 根因诊断

### 三层根因

| 根因 | 说明 |
|------|------|
| A: glm-5 太慢 | 推理模型处理 ~5K token prompt 需 2-5 分钟，思维链 reasoning_content 开销巨大，远超 120s timeout |
| B: Clash 代理长连接不稳定 | 本地代理 127.0.0.1:7897 对 >60s HTTPS 连接执行超时清理，导致 `RemoteDisconnected` |
| C: Windows 代理绕不开 | `proxies={}` 无法覆盖 Windows 注册表 `ProxyEnable=1`，仍经过 Clash |

### 对比实验

| 组合 | 短 prompt | 长 prompt (~5K tokens) |
|------|----------|----------------------|
| WITH 系统代理 | 200 OK, 11.49s | FAIL, 120s timeout |
| WITHOUT 代理 | 200 OK, 9.84s | FAIL, 123s ProxyError |

### 关键结论

**代码路径完整正确**，瓶颈仅在 LLM 外部调用链路。D3 成功案例完整证明了：Prompt 生成 → JSON 解析 → 动态注入 → Gurobi 求解 → 评分记录，全链路畅通。

## Phase⑤-G-I：代理问题修复（同日下午）

根据诊断报告，启动修复工作。经讨论，先改绕代理，暂不换模型。

### 诊断与执行

**问题分析**：
- `requests.post()` 默认通过 `urllib.getproxies()` 读 Windows 注册表
- 即使传 `proxies={}` 也会被环境设置覆盖
- 唯一可靠方案：`requests.Session(trust_env=False)`

**修改内容**：
- 文件：`llmAPI/llmInterface_huggingface.py`
- `__init__` 中新增 Session 创建：
  ```python
  self._session = requests.Session()
  self._session.trust_env = False  # 绕过 Windows 注册表代理
  ```
- `getResponse()` 中替换调用：`requests.post` → `self._session.post`
- 改动：2 处修改

### Codex 审查结论

✅ **通过**，无阻塞问题。
- ✅ 能准确绕开 Windows 代理（跳过 `urllib.getproxies()` → 注册表链路）
- ⚠️ 副作用：同时关闭 `.netrc` 自动认证和 `REQUESTS_CA_BUNDLE` 环保证书注入
- ✅ 线程安全：每个 worker 创建独立 API 对象 → 独立 Session，无跨线程共享
- ✅ 副作用对本项目无影响
