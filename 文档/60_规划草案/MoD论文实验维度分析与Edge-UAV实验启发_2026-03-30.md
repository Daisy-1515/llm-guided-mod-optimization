# MoD论文实验维度分析与 Edge-UAV 实验启发

**创建日期**: 2026-03-30  
**适用项目**: `llm-guided-mod-optimization`  
**分析对象**: *Hierarchical Optimization via LLM-Guided Objective Evolution for Mobility-on-Demand Systems*  
**目标**: 归纳原论文通过哪些实验维度证明其结论，并提炼对当前 Edge-UAV 项目实验设计的直接启发。

---

## 1. 背景与目的

原论文的贡献不只是“LLM 可以参与优化”，而是提出了一个 `LLM + Harmony Search + Optimizer + Simulator` 的闭环分层系统，并试图证明：

1. 这种混合式方法优于手工目标、RL、纯 LLM 方法。
2. 优势在大规模、长时域、动态环境下更明显。
3. 真正有效的关键不只是 LLM 本身，而是 `闭环反馈`、`结构化 prompt` 和 `约束显式化`。

本报告分两部分：

1. 总结原论文从哪些实验维度支撑上述结论。
2. 提炼这些实验设计对当前 `Edge-UAV` 项目的实验启发，形成可执行的实验维度建议。

---

## 2. 原论文核心结论

基于原文主实验、消融实验和附录补充实验，论文主要想证明以下 5 条结论：

1. `Hybrid LLM + Optimizer` 整体优于手工目标、RL 和纯 LLM 方法。
2. 方法优势在 `大规模 / 高需求 / 长时域` 场景更明显。
3. `dynamic closed-loop` 比 `single open-loop` 更有效，说明动态目标演化是关键机制。
4. `model blueprint + restriction prompt` 对性能和可行性都很关键。
5. Harmony Search 参数和 LLM temperature 会实质影响最终效果，说明该框架存在明确的优化动力学，而非偶然有效。

## 4. 各实验维度如何支撑论文结论

### 4.1 跨方法家族对比

这是论文最核心的主证据。

- 对比对象包括：
  - 手工目标函数
  - RL 方法
  - 纯 LLM 方法
  - 作者提出的 Hybrid 方法
- 主指标为 `平均乘客等待时间`
- 关键图表：
  - `Table 2`: New York
  - `Table 3`: Chicago

这组实验支撑的核心结论是：

1. 论文不是只证明“比某一个 baseline 强”，而是证明其相对三类不同方法路线都更强。
2. Hybrid 的优势来自“LLM 生成高层目标 + 优化器保证底层可行性”的结合，而不是单独依赖 LLM。

### 4.2 跨城市泛化

论文没有只在一个城市做仿真，而是分别在 `New York` 和 `Chicago` 两套出租车数据上测试。

- New York 更接近相对平衡的出行分布
- Chicago 的需求分布更集中、更不均衡

这组实验主要证明：

1. 方法不是只适配某一种空间结构。
2. 结论在不同城市拓扑与 OD 分布下仍成立。

这使论文的“泛化性”论证强于只做单城实验的工作。

### 4.3 跨规模场景

每个城市都设计了 `9` 个 scenario，通过 passenger 数、taxi 数、time horizon 逐步提升复杂度。

这组实验主要证明：

1. 论文方法在小规模场景下未必全面碾压。
2. 但在大规模、高需求、长时域场景下优势显著扩大。
3. 因此论文最强的卖点不是“小样本也最优”，而是“复杂动态场景下更稳、更能扩展”。

从学术表达上，这一点很重要，因为它决定了论文的 strongest claim 应当是：

`方法在复杂动态优化场景下更具优势`

而不是泛泛地说：

`方法在所有场景都绝对最优`

### 4.4 高压场景时空分析

论文没有停留在“均值表格”，还对最难的 `Scenario 9` 做了时空热图分析。

- 观察对象：不同方法在不同 zone、不同时间窗口下的等待时延
- 关键图表：
  - `Figure 7`: New York
  - `Figure 8`: Chicago

这组实验支撑的结论是：

1. 作者方法不仅平均等待时间更低。
2. 它还能够更快消化拥堵、减少延迟向后续时段累积。
3. 因而它的改进是“行为层面的改善”，不只是一个最终均值更好。

这类分析通常比单纯表格更有说服力，因为它回答了“为什么更好”。

## 5. 证据强弱与论文边界

### 5.1 证据最强的部分

以下结论证据链最完整：

1. `Hybrid 方法优于多类基线`
2. `复杂场景下优势更明显`
3. `闭环反馈确实有效`
4. `结构化 prompt 和约束显式化确实重要`

因为这些都有较直接的主表或消融表支撑。

### 5.2 证据相对较弱的部分

以下说法需要保守：

1. `全面提升运营质量`
   - 论文主指标几乎围绕 `平均等待时间`
   - 对 fairness、收益、能耗、司机体验等指标支撑不足

2. `真实部署就一定有效`
   - 实验仍是仿真，不是线上 A/B 测试

3. `training-free 一定优于 RL`
   - 论文展示了 RL 表现较弱
   - 但没有在统一训练预算、算力预算、数据预算下做更严格公平比较

### 5.3 对我们读论文时的启示

阅读这篇论文时，最应该学的不是某个具体数值，而是它如何把论证拆成：

1. 主结果
2. 泛化
3. 机制消融
4. 可行性
5. 稳定性

这正是一个完整实验叙事的基本骨架。

## 6. 对当前 Edge-UAV 项目的实验设计启发

原论文的实验思路对当前项目有 8 条直接启发。

### 6.1 不要只做“我的方法 vs 一个 baseline”

当前项目如果只比较：

- `LLM + HS + BCD`
- 某一个固定目标 baseline

那么证据链偏弱。

更合理的做法是按“方法家族”比较：

1. `完整方法`: LLM + HS + BCD/SCA
2. `无 LLM`: 手工固定目标 + HS + BCD/SCA
3. `无 HS`: LLM 单次或独立采样
4. `无闭环`: LLM once-per-individual
5. `无 BCD loop`: 只做 Level-1 或只做顺序一次解耦
6. `默认固定目标`: 作为经典工程基线

这样才能证明到底是哪一层机制在起作用。

### 6.2 必须做跨规模，而不是只测默认参数

你的问题天然具有规模敏感性：

- 终端数增加
- UAV 数增加
- 时隙数增加
- 空间约束和安全约束变强

因此至少应设计：

- `Small`
- `Medium`
- `Large`

必要时加一个 `Stress` 场景。

否则很难证明你的方法在复杂场景下仍有优势，也难以支撑论文中的“scale-up claim”。

### 6.3 不能只看总目标值

原论文虽然主指标偏单一，但它至少有清晰主指标。

对 Edge-UAV 项目，建议把指标拆成四类：

1. `任务性能`
   - 总时延
   - 平均时延
   - 违约率 / 超时率

2. `资源效率`
   - 计算资源利用率
   - 带宽利用率
   - UAV 服务覆盖效率

3. `物理代价`
   - 飞行能耗
   - 计算能耗
   - 总能耗

4. `算法代价`
   - 可行率
   - wall-clock time
   - 求解器调用次数
   - LLM 调用次数

如果只报一个总目标值，答辩时会很被动，因为别人会追问“改善是从哪里来的、是否牺牲了别的指标”。

### 6.4 必须把“可行率 / 错误率”独立拿出来

你的项目比 MoD 更复杂，因为存在：

- 非凸通信公式
- 轨迹变量
- BCD/SCA 近似
- LLM 生成启发式目标

因此建议把以下失败类型单独统计：

1. `LLM 输出非法`
2. `exec / parser 失败`
3. `求解器 infeasible`
4. `SCA 不收敛或数值异常`
5. `BCD 达到上限仍未稳定`

这会极大提升论文可信度，因为你不是只报告“成功样本里最好的结果”。

### 6.5 机制消融要对准你的真实创新点

对你当前项目而言，最重要的消融不是泛泛删模块，而是对应真实创新点：

1. `LLM objective` 是否优于固定代理目标
2. `HS evolution` 是否优于无演化独立采样
3. `BCD outer loop` 是否优于一次性顺序求解
4. `trajectory optimization` 是否真的给 Level-1 带来反馈收益
5. `restriction / blueprint prompt` 是否显著降低 invalid objective rate
6. `warm-start / inexact BCD` 是否显著降低计算成本

这些才是你论文的“证据抓手”。

### 6.6 增加一个高压场景的行为分析

除了总表外，建议挑一个最难场景做深入分析，例如：

- `任务时延随 time slot 变化曲线`
- `UAV 轨迹热图`
- `卸载决策分布图`
- `能耗与时延 tradeoff 曲线`
- `每轮 BCD 的 objective 改善曲线`

这类图的价值是回答：

`方法为什么更好`

而不只是：

`最终数值更低`

### 6.7 统一比较预算

如果你的方法调用 LLM、运行 BCD/SCA、做多轮 HS，而 baseline 只跑一次求解，那么比较很容易被质疑“不公平”。

建议至少统一以下一种口径：

1. `统一 evaluation budget`
2. `统一 wall-clock budget`
3. `统一 solver call budget`

并在报告中明确写出你采用的是哪一种。

### 6.8 主实验与补充实验要分层

建议直接沿用原论文的叙事结构：

1. `Main results`
2. `Ablation study`
3. `Robustness / sensitivity`
4. `Case study`
5. `Failure analysis`

这样后续写论文最顺，不会出现“数据很多但叙事很散”的问题。

## 7. 对当前项目的建议实验维度矩阵

### 7.1 必做维度

| 维度 | 推荐设置 | 目标 |
|---|---|---|
| 跨方法家族对比 | Full / no-LLM / no-HS / fixed objective / reduced BCD | 证明完整方法有必要 |
| 跨规模场景 | Small / Medium / Large / Stress | 证明可扩展性 |
| 主指标对比 | delay / energy / feasible_rate / wall_time | 证明改进不是单一幻觉 |
| 机制消融 | prompt、HS、BCD、trajectory feedback | 对齐真实创新点 |
| 错误率分析 | invalid objective / infeasible / non-convergent | 证明鲁棒性 |

### 7.2 应做维度

| 维度 | 推荐设置 | 目标 |
|---|---|---|
| 超参数敏感性 | HMCR / PAR / temperature / BCD 轮数 | 说明方法稳定区间 |
| 高压案例分析 | hardest scenario 的时序图与轨迹图 | 解释性能来源 |
| 预算公平性分析 | fixed eval budget 或 fixed time budget | 回答公平性质疑 |

### 7.3 可选维度

| 维度 | 推荐设置 | 目标 |
|---|---|---|
| 跨 prompt 结构泛化 | 不同 blueprint 粒度 | 研究 prompt grounding 机制 |
| 跨环境分布 | 更稀疏 / 更拥塞 / 更高移动约束 | 研究鲁棒性边界 |
| 跨求解策略 | exact vs inexact BCD | 研究效果-成本权衡 |

## 8. 一个适合当前项目的论文级实验叙事模板

如果后续要把实验写成论文，建议按下面的顺序组织：

### 8.1 主结果

回答：

- 你的完整方法是否优于主 baseline
- 优势在什么场景最明显

### 8.2 机制消融

回答：

- LLM 是否真正有效
- HS 是否真正有效
- BCD 反馈是否真正有效
- prompt blueprint / restriction 是否真正有效

### 8.3 稳定性与成本

回答：

- 方法是否稳定
- 算法代价是否可接受
- 成本换来的收益是否合理

### 8.4 个案分析

回答：

- 为什么会更好
- 改善发生在系统的哪个层面

### 8.5 失败案例与边界

回答：

- 在哪些场景下方法失效
- 失败来自 LLM、求解器、还是 BCD/SCA 耦合

这部分往往能显著提升论文说服力。

## 9. 推荐优先级

如果实验资源有限，建议优先级如下。

### 第一优先级

1. 主结果表
2. 跨规模场景
3. 机制消融
4. 可行率/错误率

### 第二优先级

1. 高压案例分析
2. 超参数敏感性
3. 预算公平性分析

### 第三优先级

1. 更复杂的 prompt 结构泛化
2. 更多环境分布外推
3. 更高保真仿真扩展

## 10. 最终结论

原论文最值得借鉴的不是某个具体数值，而是它的实验论证框架：

1. 用 `主结果` 证明方法有效。
2. 用 `跨城市 / 跨规模` 证明泛化。
3. 用 `消融` 证明关键机制。
4. 用 `错误率` 证明可靠性。
5. 用 `热图 / 曲线` 解释为什么有效。
6. 用 `超参数实验` 证明方法不是偶然成功。

对当前 Edge-UAV 项目而言，最重要的启发是：

`不要把实验只做成“一个总 cost 表”，而要做成“主结果 + 机制消融 + 失败分析 + 行为解释”的完整证据链。`

这会直接决定你后续论文能否把“LLM-guided optimization”讲成一个可信的学术故事，而不是一个偶然有效的工程 demo。
