## 3. 核心分析

### 3.1 BCD ON 的稳定性问题

**现象**: BCD ON 在每一代都有 1 个目标函数生成失败（80% 成功率），而 BCD OFF 为 100%。

**失败特征**:
- 始终是同一成员失败（可能是同一个体的同一轮 LLM 调用）
- 并未影响可行性（所有 5 个解都满足约束）
- 未影响最优值（同样达到 31.7735）

**可能原因**:
1. **BCD 与 LLM 的交互问题**: BCD 轨迹优化后的结果被送回 LLM 生成新目标函数，某个转换步骤失败
2. **JSON 序列化问题**: `Level2Snapshot` → JSON 转换中数值精度丧失或类型错误（Codex 提出的 Warning）
3. **LLM API 超时**: 特定个体的 prompt 过长或 API 不稳定
4. **offloading 结果验证失败**: BCD 环节产生的决策在 validation 中被拒（见 validate_offloading_outputs 的 Warning）

**影响**: 虽然不阻断优化流程（有默认目标函数），但增加了求解风险。

### 3.2 完全相同的最优值 — BCD 无效?

**现象**: 10 代中，两种配置的最优值始终为 31.7735（0% 差异）。

**分析**:

| 因素 | 说明 |
|------|------|
| **Level 1 (LLM) 效能** | LLM 生成的目标函数已足够优化，给出接近最优的解 |
| **Level 2a (BCD) 改进空间** | 问题规模小，UAV 轨迹优化空间有限（已受初值约束） |
| **初值质量** | Harmony Search 的选择机制已产生高质量个体，BCD 难以改进 |
| **问题结构** | 3 UAVs, 10 tasks, 20 slots 的问题规模，单个 depot 结构 |

**结论**:
- ✅ **BCD 轨迹优化在当前场景下不增加成本收益**
- ✅ **但也不会降低成本** — 可靠的备选方案
- ⚠️ **但稳定性较弱** — 80% vs 100% 的成功率

### 3.3 本次验证的启示

| 观点 | 结论 |
|------|------|
| **BCD 必要性** | 在小规模问题上不关键（但可作为架构演进基础） |
| **系统稳定性** | 不启用 BCD 更稳定（100% 成功率） |
| **生产建议** | 当前配置下，关闭 BCD (use_bcd_loop=false) 推荐 |
| **后续研究** | 大规模问题 (100+ tasks, 10+ UAVs) 应测试 BCD 效能 |
