## 新增指标（5 个）

计算条件：预计算时有 `snapshot` 时自动计算

| 指标 | 计算方式 | 含义 | 特殊处理 |
|------|---------|------|---------|
| `assigned_pairs` | count(f_edge > eps_freq) | 被分配对总数 | — |
| `assigned_feasible_pairs` | count(f_edge > eps_freq AND d_offload ≤ tau) | 已分配对中可行的对数 | — |
| `assigned_feasible_ratio` | assigned_feasible / assigned_pairs | **已分配对的真实可行率**（关键） | **若 assigned_pairs==0，则为 None**（不是 0.0） |
| `unassigned_pairs` | count(f_edge ≤ eps_freq) | 未分配对总数（BCD 中未被分配） | — |
| `assigned_pair_ratio` | assigned_pairs / total_pairs | 系统分配率（资源利用度） | — |

**重要说明**：eps_freq = 1e-12（数值保护常数）

**关键修正**：当 assigned_pairs==0 时，assigned_feasible_ratio = None 而非 0.0
- None 表示"无已分配对，无法计算可行率"（不做 deadline 压力判断）
- 0.0 表示"有已分配对但全部不可行"（真实 deadline 压力很大）

## 数据流与时间语义

```
BCD 迭代 k=0:
  P2: precompute(current_snapshot=初始快照) → precompute_result_0
  P3-P6: 优化更新 best_snapshot

... (更多迭代) ...

BCD 迭代 k=k*:
  P2: precompute(current_snapshot=快照k*) → precompute_result_k*
  P3-P6: 优化 best_snapshot 收敛
  [收敛检查通过，标记 converged_at_k=k*，break]

BCD 返回前：
  ├─ 若 converged_at_k is not None：
  │  └─ 重新调用 precompute(best_snapshot) → final_precompute_at_convergence
  │     （确保诊断基于收敛后的最终解，不是第 k* 轮开头的初始快照）
  │
  └─ 若未收敛（达到最大迭代）：
     └─ 使用最后一轮的 precompute_result（向下兼容）

结果保存：
  bcd_result.solution_details["final_precompute_diagnostics"]
    └─ 包含分离统计的诊断字典

种群结果 (hsIndividualEdgeUav.py)：
  └─ 自动保存 solution_details 到 JSON
     └─ discussion/<run_id>/population_result_0.json
        └─ simulation_steps["0"]["bcd_meta"]["solution_details"]
           或 simulation_steps["0"]["final_precompute_diagnostics"]
           （路径视 hsIndividualEdgeUav 的保存逻辑而定）
```

**关键点**：final_precompute_diagnostics 现在可靠地反映最终解的诊断，不是中间迭代的快照

## 诊断能力

通过新指标，可以：

### 1️⃣ 识别混淆原因
```
before BCD:  offload_feasible_ratio = 50% (5/10 对)
after BCD:   offload_feasible_ratio = 0.7% (仅 1/150 对)

新指标分解:
- assigned_pairs = 30 (被分配)
- assigned_feasible = 15 (可行)
- assigned_feasible_ratio = 50% ← 真实压力
- unassigned_pairs = 120 (未分配，f=0)

解读: 混合比例下降是因为大量未分配对被算成不可行，
      但已分配对的压力（50%）保持稳定 → 这就是真实 deadline 压力
```

### 2️⃣ 评估资源利用度
```
assigned_pair_ratio = 30/150 = 20%

说明系统仅分配了 20% 的潜在对，有优化空间
```

### 3️⃣ 指导参数调整
```
若 assigned_feasible_ratio < 30%:  deadline 压力大，调整权重
若 assigned_feasible_ratio > 70%:  压力适中，继续优化
若 assigned_pair_ratio << 100%:    分配率低，检查约束强度
```

## 关键设计决策

### 为什么用 `f_edge > eps_freq` 而不是 `f_edge > 0` 来判断已分配？
- 数值安全：避免浮点精度问题
- 一致性：与预计算中的数值保护一致（eps_freq = 1e-12）
- 清晰性：完全反映实际物理含义（接近零等同于零）

### 为什么存到 solution_details 而不是独立的诊断文件？
- 紧凑性：避免额外文件
- 可追溯：诊断数据与决策结果在同一记录
- 兼容性：无需修改现有文件保存逻辑
