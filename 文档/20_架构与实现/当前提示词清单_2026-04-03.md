## 1. 结论摘要

当前仓库中有两套提示词体系：

## 2. Edge-UAV 提示词（主链路）

### 2.1 文件位置

- `edge_uav/prompt/base_prompt.py`
- `edge_uav/prompt/mod_prompt.py`

### 2.2 基础组件（`EdgeUavPrompts`）

`base_prompt.py` 定义了 5 个核心提示词组件：

1. `prompt_scenario`：系统与任务背景
2. `prompt_static_info`：场景统计信息（运行时填充）
3. `prompt_init_level1model`：`OffloadingModel` 源码与结构说明
4. `prompt_obj_format`：LLM 输出 JSON 规范（`obj_description` + `obj_code`）
5. `prompt_cons_restriction`：Gurobi 编码约束与变量使用规则

说明：

- `set_scenario_info(tasks, uavs, time_slots)` 会写入运行期统计摘要
- 约束规则中已包含 math 非线性系数变换鼓励（如 `exp/log1p/sqrt/tanh/pow`，作用于常量系数）

### 2.3 进化策略（`EdgeUavModPrompts`）

`mod_prompt.py` 定义了 4 种生成方式：

1. `get_prompt_way1`：全新生成（random-explore）
2. `get_prompt_way2`：基于最佳个体局部改进（local-refine）
3. `get_prompt_way3`：结构性重构（structural-mutate）
4. `get_prompt_way4`：资源感知专项生成（resource-aware-specialization）

### 2.4 实际路由（运行时）

在 `heuristics/hsIndividualEdgeUav.py` 的 `getNewPrompt()` 中：

- `WAY_RANDOM` -> `get_prompt_way1`
- `WAY_MEMORY` -> `get_prompt_way2`
- `WAY_PITCH` -> `get_prompt_way3`
- `WAY_CROSS` -> `get_prompt_way4`
- `default` 或异常 -> 回退默认目标函数（不走 LLM）

### 7.1 Level-1 基础提示词：`prompt_scenario`

来源：`edge_uav/prompt/base_prompt.py` -> `EdgeUavPrompts._build_scenario()`

```text
Act as an expert in Edge Computing and UAV trajectory optimization. You are designing the primary objective function for a two-level real-time computation offloading system.

1. **System Overview**:
   - Multiple UAVs fly over a service area, each carrying an edge computing server.
   - Ground terminal devices (TDs) generate computation tasks every time slot.
   - Each task can be executed locally on the TD, or offloaded to a nearby UAV for remote execution.
   - The system operates in discrete time slots indexed by t.

2. **Optimization Structure (Two-Level Decomposition)**:
   - **Level 1 (Your Task)**: Task offloading assignment - decide which tasks execute locally and which offload to which UAV. This is a Binary Linear Program (BLP) solved by Gurobi.
   - **Level 2 (Already Solved)**: Given fixed offloading decisions, jointly optimize UAV trajectories and CPU frequency allocation via BCD + SCA.
   - The two levels alternate iteratively (Block Coordinate Descent) until convergence.

3. **Your Task**:
   Design a novel Level-1 objective function that guides Gurobi to make good offloading decisions. Your objective is a *proxy* - it replaces the standard cost function to explore creative optimization strategies. The proxy objective should:
   - Minimize a weighted combination of task completion delay and energy consumption.
   - Account for task urgency (deadline), UAV proximity, and load distribution.
   - Be compatible with Gurobi's BLP formulation (linear in binary variables).

4. **Key Requirements**:
   - Each task must be assigned exactly once within its active window (local or one UAV).
   - Only active tasks (task[i].active[t] == True) should be included in summations.
   - The objective must be expressible using Gurobi linear expressions.
   - Precomputed constants (delays, energies) are available - no need to recompute them.
```

```text
=== OffloadingModel Source ===
{offloading.py 的源码全文}

=== Model Structure ===
- ComputeTask: dataclass with D_l (local data), D_r (remote data), F (CPU cycles), tau (deadline), active[t] (per-slot activity flag).

### 7.4 Level-1 输出格式提示词：`prompt_obj_format`

来源：`edge_uav/prompt/base_prompt.py` -> `EdgeUavPrompts._build_obj_format()`

    # Define cost components (1-5 components)
    cost1 = gb.quicksum(
        self.D_hat_local[i][t] / self.task[i].tau
        * self.x_local[i, t]
        for i in self.taskList for t in self.timeList
        if self.task[i].active[t]

    objective = gb.quicksum(w*c for w, c in zip(costs, weights))
    self.model.setObjective(objective, gb.GRB.MINIMIZE)"}

Requirements:
1. **NO EXPLANATIONS**: Only provide the JSON object - no markdown, no analysis.
2. **Indentation**: Every line after `def` MUST start with exactly 4 spaces.
3. **Validation**: Test your code mentally for valid Python before responding.
4. JSON must contain exactly two keys: obj_description (string) and obj_code (code block).
```

### 7.5 Level-1 约束提示词：`prompt_cons_restriction`

来源：`edge_uav/prompt/base_prompt.py` -> `EdgeUavPrompts._build_cons_restriction()`

该段很长，是约束输出格式、变量命名、active-task gating、Gurobi 表达式规则和系数多样性建议的核心来源。原文如下：

```text
5. Code implementation requirements:
   - Number of cost components: 1 <= n <= 5.
   - Weights list must match the length of costs list.
   - Final objective must use: gb.quicksum(w*c for w, c in zip(costs, weights)).

6. Variable naming (MUST use exactly):
   - self.x_local[i, t]          (binary: 1 = execute locally)
   - self.x_offload[i, j, t]     (binary: 1 = offload task i to UAV j at time t)
     Index order: i=task, j=UAV, t=time slot. DO NOT swap.
   - self.D_hat_local[i][t]       (precomputed local delay, scalar constant)
   - self.D_hat_offload[i][j][t]  (precomputed remote delay, scalar constant)
   - self.E_hat_comp[j][i][t]     (precomputed edge energy, scalar constant)
   - self.task[i].tau             (deadline, scalar constant)
   - self.task[i].active[t]       (Python bool - use in `if` guards)
   - self.uav[j].E_max           (max energy budget, scalar constant)
   - self.alpha                   (delay weight, scalar constant)
   - self.gamma_w                 (energy weight, scalar constant)
   - self.M                       (big-M constant for linearization)

7. Active-task gating (CRITICAL):
   - Every summation over tasks MUST filter by self.task[i].active[t].
   - Use Python `if` inside generator: `gb.quicksum(... for i in self.taskList for t in self.timeList if self.task[i].active[t])`.
   - NEVER sum over inactive tasks - they have no valid precomputed constants.

8. Full time-slot summation:
   - When computing per-task or per-UAV aggregates, always sum across ALL t in self.timeList.
   - Combine with the active-task filter: `for i in self.taskList for t in self.timeList if self.task[i].active[t]`.
   - Do NOT optimize for a single time slot unless explicitly instructed.

9. Scale awareness:
   - Delays are in seconds (typically 0.001 ~ 1.0 s).
   - Energy is in Joules (typically 0.1 ~ 100 J).
   - Use self.alpha and self.gamma_w to balance them, or normalize by self.task[i].tau and self.uav[j].E_max respectively.

10. Deadline awareness:
   - Consider adding a soft penalty for tasks close to their deadline.
   - Example: penalty = max(D_hat_offload[i][j][t] - task[i].tau, 0) (use Python max() since operands are constants).

11. gb.quicksum rules:
   a) NEVER nest gb.quicksum() inside another gb.quicksum().
   b) Flatten multi-dimensional sums into a single gb.quicksum with multiple `for` clauses.
      - Valid:   gb.quicksum(expr for i in ... for j in ... for t in ...)
      - Invalid: gb.quicksum(gb.quicksum(...) for i in ...)
   c) Do NOT use gb.quicksum for a single term - just use the expression directly.

12. Expression function rules:
   - Use gb.quicksum()/gb.max_()/gb.abs_() ONLY when the expression contains Gurobi variables.
   - Use Python sum()/max()/abs() when working with pure constants (parameters).
   - You may use `import math` and apply math.exp(), math.log(), math.log1p(), math.sqrt(), math.pow(), math.tanh() on PURE SCALAR CONSTANTS to create nonlinear coefficient transformations. The result is still a scalar, so multiplying it by a binary variable remains linear for Gurobi.
   - Example: `math.exp(D_hat_offload[i][j][t] / task[i].tau) * x_offload[i, j, t]` is valid — exp() operates on a constant, producing a scalar coefficient.

13. Quadratic term rules:
   - For squared terms, use variable * variable directly.
   - NEVER use Python list.append() with Gurobi expressions.

14. Conditional logic handling:
   - NEVER use Python if/else on Gurobi variables/expressions.
   - For conditional cost components that depend on binary variables, multiply by the binary variable directly (big-M linearization).

15. Nonlinear handling:
   - For products of two binary variables: create auxiliary variable + big-M constraints.
   - All precomputed constants (D_hat_*, E_hat_*) are scalars - multiplying them by a binary variable is always linear.

16. Coefficient diversity — STRONGLY ENCOURAGED:
   - Go beyond simple linear ratios (D/tau, E/E_max). Transform scalar constants with nonlinear functions to reshape the cost landscape:
     * Exponential urgency:  math.exp(min(k * D_hat / tau, 3.0))  — k=1.0~2.0 recommended
       (exp(3)≈20, exp(5)≈148, exp(10)≈22000 causes numerical instability)
     * Logarithmic saturation:  math.log1p(E_hat / E_max)  — diminishing returns on energy cost
     * Square-root smoothing:  math.sqrt(D_hat / tau)  — gentle sub-linear penalty
     * Sigmoid-like focus:  math.tanh(k * (D_hat/tau - threshold))  — output in [-1,1], safe
     * Power-law:  math.pow(D_hat / tau, p)  — p=1.5~3.0 recommended, avoid extreme values
   - These all produce scalar coefficients, so `f(constant) * binary_var` is linear for Gurobi.
   - Combine different transforms across cost components for richer trade-off surfaces.
   - CRITICAL: Keep coefficients in [0.01, 100] range. Values outside [1e-6, 1e6] will cause Gurobi numerical issues and BCD solver failures.

17. Time-window urgency — OPTIONAL but RECOMMENDED:
   - Each task i has an active window: a contiguous set of time slots where it can be served.
   - A task with a shorter active window is harder to schedule (fewer opportunities).
   - You MAY scale per-task penalty terms by a window-tightness factor computed as pure Python:
       window_size_i = sum(1 for t in self.timeList if self.task[i].active[t])  # scalar int
       tightness_i   = len(self.timeList) / max(window_size_i, 1)               # scalar float
   - Then multiply inside gb.quicksum, e.g.:
       math.log1p(tightness_i) * D_hat_local[i][t] / task[i].tau * x_local[i,t]
   - window_size_i and tightness_i are pure constants — Python arithmetic is safe here.
   - The scenario statistics section shows the distribution of tight-window tasks.
```

--- Resource-Aware Guidance ---
Pay special attention to the following resource constraints:

   - Urgent tasks (small tau) should prefer the nearest UAV even if it costs more energy.

### 7.11 Level-2b 轨迹优化提示词：公共头与规则

来源：`edge_uav/prompt/traj_prompt.py` -> `TrajectoryPrompts._header()` 与 `TrajectoryPrompts._rules()`

公共头：

```text
=== Edge-UAV Level-2b Trajectory Optimization ===

## Background
You are designing the objective function for a UAV trajectory optimizer.
The optimizer uses Successive Convex Approximation (SCA) to find UAV positions
over T time slots that minimize a weighted combination of:
  1. Communication delay  — how long it takes to upload/download task data
     (UAVs closer to their assigned tasks → lower delay)
  2. Propulsion energy    — energy spent flying the UAVs
     (UAVs moving more → higher energy)

## Pre-computed objective components (already normalized)
These are CVXPY expressions. You MUST NOT call any CVXPY methods on them;
only multiply them by positive Python float coefficients.

  obj_comm_surrogate  : convex surrogate for total normalized communication delay
                        (smaller → UAVs are closer to tasks → better offloading)
  obj_propulsion      : total normalized propulsion energy across all UAVs
                        (smaller → UAVs move less)
  obj_slack           : constraint violation penalty (MUST always be included)

## Available scalar parameters
  alpha     = {alpha}   (delay weight from config)
  lambda_w  = {lambda_w}   (propulsion weight from config)
  N_act     = {N_act}    (number of active offload pairs this BCD iter)
  N_fly     = {N_fly}    (number of UAV movement segments)
  n_uavs    = {n_uavs}
  n_tasks_active = {n_tasks_active}
```

规则：

```text
## Coding Rules (STRICT)
1. Output ONLY a JSON object: {"traj_obj_description": "...", "traj_obj_code": "..."}
2. traj_obj_code MUST assign `dynamic_traj_objective` (a CVXPY expression).
3. Coefficients MUST be positive Python floats (ensures convexity).
   VALID:   2.0 * alpha * obj_comm_surrogate
   INVALID: -alpha * obj_comm_surrogate   ← negative breaks DCP
4. You may use `import math` and math.* functions ONLY on scalar values,
   NOT on CVXPY expressions.
   VALID:   math.exp(0.5) * alpha * obj_comm_surrogate
   INVALID: cp.exp(obj_comm_surrogate)   ← may not be DCP
5. obj_slack MUST always appear with a positive coefficient (≥ 1.0).
6. Do NOT import cvxpy or gurobipy. Do NOT define functions.
7. Keep coefficients in [0.01, 100] to avoid numerical issues.

## Example traj_obj_code
<example>
import math
# Emphasize communication for many active offloads
w_comm = math.log1p(N_act / 5.0 + 1.0) * alpha
dynamic_traj_objective = w_comm * obj_comm_surrogate + 0.3 * lambda_w * obj_propulsion + obj_slack
</example>
```

### 7.15 实际发送到 LLM 的消息格式

来源：`llmAPI/llmInterface_huggingface.py` -> `prepare_payload()`

当前实现不是 `system` + `user` 双消息结构，而是将完整拼接后的 prompt 作为单条 `user` 消息发送：

```json
{
  "messages": [
    {
      "role": "user",
      "content": "完整拼接后的 prompt 文本"
    }
  ],
  "model": "..."
}
```

这意味着：

- `base_prompt.py` 和 `mod_prompt.py` 中的所有文本最终都会合并成一条长 `user` prompt。
- 代码中没有单独的 `system_message` 常量。
- 若后续需要把 `P_sce / P_model / P_dyn / P_op / P_res` 明确拆成多消息结构，需要同步改 `llmAPI` 侧请求格式。
