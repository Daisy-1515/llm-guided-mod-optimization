# BCD 多轮迭代失效根因诊断与参数设计讨论

**日期**：2026-04-06  
**方法**：Codex（代码层面）+ Gemini（系统理论）双视角联合分析  
**问题**：如何让 LLM 优化有效果 + BCD 循环也有效果？

---

## 核心结论

**调参之前，必须先修复两个代码 Bug。没有代码修复，参数调什么都无效。**

---

## 第一部分：代码层面根因（Codex 分析）

### Bug 1：`adapt_f_edge_for_snapshot()` 锁死候选空间（最关键）

**位置**：`edge_uav/bcd_loop.py` ~L303

**问题**：上一轮 L1 未选中的 `(j,i,t)` 卸载边，`f_edge` 被写成 `0.0`。下一轮 `precompute` 用零频率计算 `D_hat_offload`，结果是 `big_m_delay`（近乎不可行），导致 L1 下一轮几乎不可能把"上一轮本地"的任务翻成"本轮卸载"。

**后果**：
- L1 决策空间单调收缩，无法真正多轮迭代
- "全本地"一旦出现就变成**吸收态**（永久锁死）
- 无法把任务从 UAV j1 改派到 j2

**修复方案**：未选中边保留保底值（如 `f_max / N_tasks`），而不是置零。

### Bug 2：收敛判据用 `best_cost` 而非 `cost_new`

**位置**：`edge_uav/bcd_loop.py` ~L688

**问题**：
```python
cost_history.append(best_cost)  # 当前代码（错误）
```
记录的是历史最优值，不是当轮真实成本。只要第二轮没有继续改进，`relative_gap = 0`，立刻 break。BCD **最早在第 2 轮就会停止**，即使 L1/L2 决策还在变化。

**修复方案**：
```python
cost_history.append(cost_new)  # 应改为当轮真实成本
```

### Bug 3：`converged` 返回值不可信

**位置**：`edge_uav/bcd_loop.py` ~L735

```python
converged = (len(cost_history) < max_bcd_iter + 1)  # 几乎恒真
```

此判断不可信，外层代码不应依赖该返回值。

### 附加发现：alpha 的实际权重被高估

代码中通信项系数 = `alpha / N_act`，飞行项系数 = `lambda_w / N_fly`。

在 `|U|=3, T=80` 配置下 `N_fly≈237`，`N_act` 通常远大于 237。真实比例：

```
通信权重 / 飞行权重 = (alpha/N_act) / (lambda_w/N_fly) = alpha × N_fly / N_act
```

当 `alpha=35`，`N_fly=237`，`N_act=500` 时，实际倍数仅 ~16.6，而非 35。

## 第二部分：系统理论约束（Gemini 分析）

### 关键不等式约束

#### (a) 卸载压力约束（非平凡性）
```
F_max > τ × f_local
```
当前验证：`2e8 > 0.5 × 3e8 = 1.5e8` ✓（但裕度仅 33%）

#### (b) 通信可行性约束
```
t_comm + t_comp_uav < τ
```
**风险**：当 `D_l` 较大、UAV 距离不为 0 时，最大任务即使 UAV 在正上方也可能无法在 τ 内完成卸载。`τ=0.5s` 对部分大任务而言窗口过紧。

#### (c) 轨迹敏感性约束（BCD 耦合条件）
```
Δt_comm = D_l/B × [1/log2(1+SNR_min) - 1/log2(1+SNR_max)] ~ O(τ)
```
近端/远端通信时延差必须与 τ 同阶，否则 UAV 位置对 L1 决策无影响，BCD 解耦。

当 B 过大时，时延差趋近于 0，轨迹优化失去意义。

#### (d) SCA 目标平衡（归一化）
```
α × Σ(Delay) ≈ λ_w × ∫P_flight(v)dt
```
当前：飞行能量 ~2000J，总通信时延 ~5s，`alpha=35` → `α×delay = 175 << 2000`。

**这解释了 UAV 走直线的根因**：飞行能耗项主导 L2 目标，轨迹无动力偏向卸载任务位置。

#### (e) LLM 梯度存在性
LLM 多样性有效的前提：**必须有多个可行卸载方案**（不同权重 w 导致不同 Gurobi 最优解）。

若约束 (b) 太紧，只有 0 或 1 个可行方案，LLM 权重无意义，HS 看到零方差适应度，无法进化。

### 三层系统同时有效的必要条件

| 层 | 条件 | 验证方式 |
|----|------|---------|
| HS 进化 | `Var(Fitness(Prompts)) > ε` | LLM 权重变化 → 不同 Gurobi 整数解 |
| BCD 收敛 | `∂Delay/∂Position` 足够大 | 轨迹变化 → 显著改变卸载可行集 |
| 数值稳定 | `α/λ_w < 1e5` | SCA Hessian 不病态 |

## 第三部分：推荐参数调整

> **前提**：代码 Bug 修复后再调参。

| 参数 | 当前值 | 建议值 | 理由 |
|------|--------|--------|------|
| `alpha` | 35 | **200~500** | 补偿 `N_fly/N_act` 缩减，使通信项与飞行项均衡 |
| `B_up / B_down` | 2e7 | **5e6~1e7** | 降低带宽，UAV 位置对通信时延影响更显著 |
| `tau` | 0.5s | **1.0~1.5s**（可选） | 扩大卸载可行性窗口，降低"必须卸载"任务卡住的概率 |
| `F_max` | 2e8 | **3e8~4e8**（随 τ 调整） | 保持 25%~40% 任务处于"边际卸载区" |

---

## 优先级排序

```
① 修复 adapt_f_edge_for_snapshot()：未选中边保留保底 f_edge
  → 解除决策空间单调收缩，让 BCD 能真正多轮翻转决策

② 修复 cost_history.append(cost_new)
  → 让收敛判断基于当轮真实成本，而非历史最优

③ 提高 alpha → 200~500
  → 让轨迹优化有动力偏向通信任务位置

④ 降低 B → 5e6~1e7
  → 通信时延对 UAV 位置更敏感，BCD 耦合更强

⑤ 可选：提高 tau → 1.0s
  → 扩大可行性窗口（需重新评估其他约束）
```
