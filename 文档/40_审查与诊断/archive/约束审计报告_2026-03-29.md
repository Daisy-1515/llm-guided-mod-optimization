## 1. 结论摘要

当前系统可稳定给出求解输出，但存在多处“部分实现”约束，导致“流程可解”与“任务语义成功”不完全等价。最关键问题是：

1. `L1-C2` 只对卸载链路做了时延硬筛选，本地时延未做硬约束。
2. `L2-C4` 安全距离采用 slack 软约束，不是严格硬约束。
3. 资源与能量耦合可行性校验多数为检查告警，不是强制拦截。

## 3. 关键偏差说明

### 3.1 L1 截止时延约束“单侧硬化”

- 当前仅对卸载链路执行 `D_hat_offload <= tau` 才允许建 `x_offload`。
- `x_local` 对 active 任务总是可选，且 `x_local + sum(x_offload) == 1` 强制必须选一个。
- 结果：当“全部 UAV 不可卸载且本地也超时”时，模型仍可能被迫选本地，语义上是失败任务，但流程层面仍可求解。

### 3.2 安全距离是软约束

- 轨迹子问题中安全距离采用线性化 + slack + 罚项。
- 这提升了数值鲁棒性，但不保证每轮都严格满足原始硬安全距离。

### 3.3 能量耦合约束未形成强拦截

- BCD 内对资源可行性主要做“有限性/非负性”检查。
- 检查不通过时记录 warning，流程继续，而非强制回退或报错中止。

### 3.4 BCD 外层成本仍为代理实现

- 当前 `P7` 注释写明“完整系统成本计算待实现”，暂以轨迹成本代理。
- 这会削弱“真实目标一致性”与回滚机制的判别力。
- 证据：`edge_uav/model/bcd_loop.py:618-622`

## 4. 风险与现象

1. 全局 `feasible=true` 不等于所有 active `(i,t)` 都满足业务成功语义。  
2. 在高负载/紧时延场景中，任务级失败可能被“代价吸收”而非显式报告。  
3. LLM 目标优化可能出现“看似有效但语义收益弱”的情况（目标函数与真实完成语义存在间隙）。

## 6. getOutputs() 输出层补充说明（Phase⑦ 补救）

**更新日期**：2026-03-30

### 背景

在 Phase⑦ 的 L1 输出层完整性补救中，扩展了 `AssignmentModel.getOutputs()` 的输出格式，新增 `drop` 字段用于显式标记被强制丢弃的任务。

### drop 字段的定义与语义

```python
drop = [
    {
        'task_id': i,
        'timeslot': t,
        'reason': 'both_infeasible',
        'local_deadline_exceeded': True,
        'best_offload_deadline_exceeded': True,
        'cost_excess': float
    },
    ...
]
```

#### 字段说明

| 字段 | 类型 | 含义 |
|------|------|------|
| `task_id` | int | 任务编号（$i$） |
| `timeslot` | int | 时隙编号（$t$） |
| `reason` | str | 丢弃原因标签，当前为 `'both_infeasible'` |
| `local_deadline_exceeded` | bool | 本地执行时延是否超过截止时间（$D_{i,i}^t > \tau_i$） |
| `best_offload_deadline_exceeded` | bool | 所有 UAV 中最优卸载选项时延是否超过截止时间（$\min_j D_{i,j}^t > \tau_i$） |
| `cost_excess` | float | 失败罚项系数，定义为 $|D_{\text{best}} - D_{\text{local}}|$（两种执行方式的成本差） |

### drop 字段的创建条件

drop 记录仅在以下条件**同时满足**时创建：

1. 任务 $(i,t)$ 在当前活跃集中（active only）
2. **本地路径不可行**：$D_{i,i}^t > \tau_i$
3. **卸载路径全不可行**：$\min_j \{D_{i,j}^t\} > \tau_i$

**结果**：任务被迫选择本地执行（通过约束 $x_{i,i}^t + \sum_j x_{i,j}^t = 1$ 强制），但实际上是失败任务，需要显式标记。

### drop 字段的用途

1. **任务级可观测性**：明确显示哪些任务因为时延硬约束而被迫放弃
2. **成本透明化**：记录失败罚项系数，用于后续成本分析
3. **约束严格化验证**：确认 L1-C2 时延约束确实被执行，而不是被隐式吸收
4. **LLM 目标反馈**：提供任务失败的详细信息，辅助 LLM 优化目标函数权重

### getOutputs() 输出格式示例

```python
outputs = solver.getOutputs()
# outputs 现在包含：
# {
#     'x': ...,  # 卸载决策矩阵
#     'feasible': bool,
#     'drop': [
#         {'task_id': 2, 'timeslot': 1, 'reason': 'both_infeasible', ...},
#         {'task_id': 5, 'timeslot': 2, 'reason': 'both_infeasible', ...}
#     ]
# }

# 后续处理中可以统计：
drop_count = len(outputs['drop'])
if drop_count > 0:
    print(f”警告：{drop_count} 个任务失败”)
    for rec in outputs['drop']:
        print(f”  Task {rec['task_id']}@t{rec['timeslot']}: cost_excess={rec['cost_excess']:.2f}”)
```

### 向下兼容性

- 若 `drop = []`（空列表），则所有活跃任务都找到可行的执行方案
- drop 字段为 **可选**（Level 2 不强制依赖），不破坏现有评分链路
- Level 2 验证和评分流程保持原样，drop 作为诊断信息被保存但不影响决策

### 后续修复方向

当前 drop 仅标记任务失败，后续可扩展为：
- 添加 `recovery_option`：是否可通过参数调整而恢复
- 记录 `failure_margin`：超期多少时间单位
- 支持多种失败原因（当前仅 `both_infeasible`，将来可能增加资源耗尽等）
