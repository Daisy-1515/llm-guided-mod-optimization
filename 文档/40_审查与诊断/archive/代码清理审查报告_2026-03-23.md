## 二、确认修复的 7 个问题

### Fix #1：共享 PrecomputeResult 跨种群个体 [HIGH — 效率]

**影响文件**：`heuristics/hsPopulation.py`、`heuristics/hsIndividualEdgeUav.py`、`edge_uav/prompt/base_prompt.py`

**现状**：
`hsIndividualEdgeUav.__init__`（第 91-96 行）每次创建个体时都执行：
```python
self.params = PrecomputeParams.from_config(configPara)
self.snapshot = make_initial_level2_snapshot(scenario)
self.precompute_result = precompute_offloading_inputs(scenario, self.params, self.snapshot)
```
而 `hsPopulation` 使用 `ThreadPoolExecutor` 并行创建 `popSize` 个个体（第 50-62 行）。每个个体的 `configPara` 和 `scenario` 完全相同，因此 `precompute_offloading_inputs`——涉及 `|tasks| × |uavs| × |time_slots|` 的三重循环加通信速率计算——被重复执行了 `popSize` 次（当前配置 3 次，生产配置可能 10+）。

此外，每个个体还通过 `EdgeUavModPrompts.__init__` → `base_prompt._read_model_source` 重新从磁盘读取 `offloading.py` 源码（第 80-85 行）。该文件在运行期间不会变化。

**为什么要修**：
- 预计算是 Phase⑥ 中最昂贵的初始化操作。Phase⑥ 新增 resource_alloc 和 trajectory_opt 后，预计算复杂度还会增长。不修复意味着冗余成本随模型复杂度线性增长。
- 磁盘 IO 在 ThreadPoolExecutor 并行场景下会产生不必要的文件句柄竞争。

**修复方案**：
- `hsPopulation.__init__` 中（仅 edge_uav 模式）预计算一次，存为 `self._shared_precompute`。
- `hsIndividualEdgeUav.__init__` 新增 `shared_precompute=None` 关键字参数：传入时直接引用，不传时回退到自行计算（向后兼容）。
- `base_prompt._read_model_source` 改用 `@lru_cache` 缓存磁盘读取。

**线程安全分析**（Codex 交叉审查确认）：
- `PrecomputeResult` 是 `@dataclass(frozen=True)`，顶层引用不可变。
- 内部 `D_hat_local`、`D_hat_offload`、`E_hat_comp` 是嵌套 dict，理论上可变。但全代码路径中，消费方（`offloading.py:202`、`offloading.py:315`、`evaluator.py:158`）均为只读访问，未发现任何写路径。
- `OffloadingModel` 和 `InterfaceAPI` 实例保持 per-individual，不共享——这是正确的。
- **结论：线程安全，可以共享。**

### Fix #2：复用物理计算函数（channel gain / Shannon rate）[MEDIUM — 复用]

**影响文件**：`edge_uav/scenario_generator.py`、`edge_uav/model/precompute.py`

**现状**：
`scenario_generator.py` 的 `_estimate_offload_delay_lb`（第 320-356 行）独立实现了信道增益和 Shannon 速率计算：
```python
dist_sq = max(dx*dx + dy*dy + h_sq, 1e-12)
gain = float(config.rho_0) / dist_sq
r_up = float(config.B_up) * math.log2(1.0 + float(config.P_i) * gain / float(config.N_0))
```

而 `precompute.py` 的 `_channel_gain`（第 532-545 行）和 `_rate_from_gain`（第 548-561 行）是经过审查的标准实现，使用 `math.log1p` 保证数值稳定性，并有 `eps_rate` 下界守卫。

**为什么要修**：
- **公式不一致**：scenario_generator 用 `math.log2(1 + x)`，precompute 用 `log1p(x) / ln(2)`。前者在 `x` 极小时有浮点精度损失。目前影响有限（仅用于场景可行性检查），但 Phase⑥ 的 resource_alloc 和 trajectory_opt 都会需要同样的物理计算，如果各自实现就会有 4 份不同版本。
- **Phase⑥ 防线**：现在统一到一处，Phase⑥ 新模块直接导入即可，不用再写第三份。

**修复方案**：
- `scenario_generator.py` 直接导入 `precompute._channel_gain` 和 `_rate_from_gain`。
- Codex 指出导入 `_` 前缀私有函数不够稳，但考虑到它们都在 `edge_uav/model/` 同一包内，且 Phase⑥ Step 3 计划创建 `physics.py` 公共模块时会重构，当前方案可接受。

### Fix #5：提升 `math.log(2.0)` 为模块级常量 [LOW — 效率]

**影响文件**：`edge_uav/model/precompute.py`

**现状**：
`_rate_from_gain`（第 558 行）每次调用都计算 `math.log(2.0)`：
```python
rate = bandwidth * math.log1p(snr) / math.log(2.0)
```
该函数在预计算中被调用 `|tasks| × |uavs| × |time_slots| × 2`（上行+下行）次。以 50 tasks × 3 UAVs × 20 slots = 6,000 次调用计算，`math.log(2.0)` 被重复计算 6,000 次。

**为什么要修**：
- 虽然 `math.log` 本身很快（C 层面），但常量提升是零风险、零副作用的优化，且让代码意图更清晰（"这是一个常数"）。
- Phase⑥ 会增加预计算复杂度，调用次数可能翻倍。

**修复方案**：
```python
_LN2 = math.log(2.0)
```
模块级定义，`_rate_from_gain` 中引用 `_LN2`。

### Fix #6：为自定义 `__eq__` 的类显式声明 `__hash__ = None` [LOW — 质量]

**影响文件**：`edge_uav/data.py`

**现状**：
`ComputeTask`（第 15 行起）和 `UAV`（第 75 行起）都定义了 `__eq__`，但未定义 `__hash__`。

Python 3 规则：定义了 `__eq__` 但未定义 `__hash__` 的类，`__hash__` 被隐式设为 `None`，实例不可哈希。这意味着 `set({task})` 或 `{task: value}` 会抛 `TypeError`。

**为什么要修**：
- 当前行为已经正确（不可哈希），但是隐式的。显式声明 `__hash__ = None` 是 Python 社区惯例，让代码审查者和 IDE 都能明确意图。
- Codex 指出：如果要动 data.py，更有价值的是补全 `__eq__`（缺少 `active`、`f_local`、`N_max`）。但补全 `__eq__` 可能影响现有测试的行为，不属于 simplify 范围，记录为后续改进项。

**修复方案**：
在 `ComputeTask` 和 `UAV` 类体中各加一行 `__hash__ = None`。

### Fix #7：统一 `initialize_population` / `generate_new_population` [LOW — 质量]

**影响文件**：`heuristics/hsPopulation.py`

**现状**：
两个方法（第 50-62 行 vs 第 64-77 行）结构完全相同：
```python
def initialize_population(self):
    results = []
    try:
        with ThreadPoolExecutor(max_workers=self.num_threads) as executor:
            futures = [executor.submit(self.get_init_ind) for _ in range(self.popsize)]
            results = [future.result() for future in as_completed(futures, timeout=self.timeout)]
    except Exception as e:
        raise RuntimeError(f"initialize_population failed: {e}") from e
    return results

def generate_new_population(self, pop):
    results = []
    try:
        with ThreadPoolExecutor(max_workers=self.num_threads) as executor:
            futures = [executor.submit(self.get_new_ind, pop) for _ in range(self.popsize)]
            results = [future.result() for future in as_completed(futures, timeout=self.timeout)]
    except Exception as e:
        raise RuntimeError(f"generate_new_population failed: {e}") from e
    return results
```
唯一区别：submit 的目标函数和参数不同。

**为什么要修**：
- DRY 原则。如果需要修改并行策略（如超时处理、异常收集、进度回调），当前需要改两处且保持一致。
- Phase⑥ 可能需要调整并行逻辑（如 BCD 循环内的子种群），统一后只改一处。

**修复方案**：
提取 `_run_parallel(self, fn, *args)` 私有方法，两个公开方法变为一行调用。保持 `as_completed` 语义不变（Codex 提醒不要换成 `executor.map`，后者会改变返回顺序）。
