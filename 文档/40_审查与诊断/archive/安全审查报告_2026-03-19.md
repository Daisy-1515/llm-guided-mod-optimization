### 疑似问题 1：`exec()` 动态代码执行

- **文件**: `edge_uav/model/offloading.py:97-107`
- **类别**: 代码注入（CWE-95）
- **初始置信度**: 0.95
- **最终判定**: 误报（最终置信度 2/10）

**数据流追踪**：

```
hsIndividualEdgeUav.getNewPrompt()    [line 218]
  → LLM API (requests.post)            [llmInterface_huggingface.py:108]
  → _parse_content() 3路解析            [llmInterface_huggingface.py:56-95]
  → extract_code_hsIndiv()              [hsIndividualEdgeUav.py:249]
  → OffloadingModel(dynamic_obj_func=)  [hsIndividualEdgeUav.py:271]
  → exec(code, globals(), namespace)    [offloading.py:100]
```

**排除理由**：

1. **既有架构模式**：与 `model/two_level/AssignmentModel.py:32-46` 完全一致，是项目三层架构核心设计（NeurIPS 论文）
2. **可信输入源**：API 端点通过环境变量 `HUGGINGFACE_ENDPOINT` 配置（环境变量为可信值，见排除先例 #3）
3. **无不可信输入路径**：无 Web 端点、无 CLI 用户输入、无外部数据流入 `exec()`
4. **缺乏加固 ≠ 漏洞**：沙箱缺失属于防御性编程建议（排除规则 #7），非具体可利用漏洞

---

### 疑似问题 2：`exec()` 中 `globals()` 暴露完整命名空间

- **文件**: `edge_uav/model/offloading.py:100`
- **类别**: 命名空间污染
- **初始置信度**: 0.85
- **最终判定**: 误报（最终置信度 2/10）

**排除理由**：与问题 1 相同——`exec(code, globals(), namespace)` 是与原有 `AssignmentModel.py` 一致的既有架构模式。攻击前提（控制 API 端点返回内容）需要攻击者控制运维人员自行配置的环境变量，不构成应用层漏洞。
