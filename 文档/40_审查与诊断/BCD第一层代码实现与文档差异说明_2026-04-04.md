- 日期：2026-04-04

- 目的：说明当前 BCD 第一层的真实代码实现，并明确它与现有文档口径的具体差异

## 1. 结论摘要

当前项目中的 BCD 第一层，即 Level 1 / offloading，已经不是“L1-C2 硬约束 + drop 失败机制”的版本，而是“无 drop、超时通过目标函数自然惩罚”的版本。

当前最接近代码现状的依据应以以下文件为准：

- `edge_uav/model/offloading.py`
- `edge_uav/model/objectives.py`
- `tests/test_l1_completeness.py`
- `文档/10_模型与公式/代码-公式差异记录_2026-04-03.md`

## 2. 第一层在代码中的位置

BCD 主循环中的第一层入口位于 `edge_uav/model/bcd_loop.py`：

1. 先基于当前快照执行 `precompute_offloading_inputs()`，生成 Level 1 所需的 `D_hat_local`、`D_hat_offload`、`E_hat_comp`
2. 再构造 `OffloadingModel`
3. 调用 `solveProblem()` 求解 Level 1 卸载决策
4. 用 `validate_offloading_outputs()` 检查输出格式

这意味着第一层在当前实现里是“预计算常数 + 二元分配模型”的结构。

---

## 3. 当前代码的真实实现

### 3.1 变量创建

`edge_uav/model/offloading.py` 的 `setupVars()` 为所有活跃 `(i,t)` 无条件创建：

- `x_local[i, t]`
- `x_offload[i, j, t]`

当前并不会因为 `D_hat_local` 或 `D_hat_offload` 超过 `tau` 而跳过变量创建。

### 3.2 约束

当前 `setupCons()` 实际只建立两类约束：

- `L1-C1`：每个活跃 `(i,t)` 必须且只能分配一次
- `L1-C3`：若 `uav.N_max` 非空，则添加 UAV 每时隙承载上限

当前代码中没有把 `L1-C2` 建成模型硬约束。

### 3.3 目标函数

当前默认目标函数位于 `default_dynamic_obj_func()`，只有两项：

- `cost1`：归一化时延成本
- `cost2`：归一化边缘计算能耗成本

即：

```text
minimize = cost1 + cost2
```

当前没有 `cost3`，也没有 `penalty_drop`。

### 3.4 输出格式

`getOutputs()` 当前返回格式为：

```python
{t: {"local": [...], "offload": {j: [...]}}}
```

当前不包含 `drop` 字段。

### 3.5 `_local_feasible` / `_offload_feasible` 的地位

这两个方法仍保留在 `offloading.py` 中，但当前代码未用它们筛变量，也未据此建立 L1-C2 约束。它们更接近诊断辅助接口，而不是当前求解逻辑的一部分。

## 4. 与现有文档的具体差异

### 4.1 差异一：L1-C2 已不是硬约束

`文档/10_模型与公式/公式与两层解耦整合版_最新版_2026-03-24.md` 中的 Level 1 仍写作：

- 卸载超时应被禁止
- 本地超时也应检查
- 全部不可行时走失败任务语义

但当前代码并未执行这套逻辑。当前行为是：

- 超时选项仍可被创建和选择
- 超时会通过 `D_hat / tau > 1` 让目标值变差
- 这是软惩罚，不是硬禁止

### 4.2 差异二：drop 失败机制已移除

旧文档里仍保留以下口径：

- `drop` 变量存在
- 目标函数存在 `cost3`
- `getOutputs()` 返回 `"drop"` 字段

当前代码全部不再成立：

- `drop` 变量不存在
- `cost3` 不存在
- 输出格式不含 `"drop"`

### 4.3 差异三：代码-文档一致性验证报告引用的是旧实现

`文档/40_审查与诊断/代码-文档一致性验证报告_2026-03-30.md` 中关于第一层的关键结论，包括以下内容，已经不再匹配当前代码：

- “L1-C2 已完整修复”
- “drop 机制存在”
- “getOutputs 包含 drop 字段”
- “目标函数为 cost1 + cost2 + cost3”

这份报告描述的是 2026-03-30 的实现状态，不是 2026-04-01 之后的现状。

### 4.4 差异四：L1-C3 基本一致

关于 `L1-C3`，代码与文档口径基本一致：

- 文档写法：可选约束
- 当前实现：仅当 `N_max` 非空时添加

这里不存在核心语义冲突。

## 5. 当前第一层应如何表述

如果要基于当前代码重新描述第一层，建议统一成下面的说法：

1. Level 1 是一个二元分配模型，决策变量为 `x_local[i,t]` 与 `x_offload[i,j,t]`
2. 对每个活跃 `(i,t)`，必须且只能选择本地或某架 UAV
3. 若配置了 `N_max`，则每个 UAV 在每个时隙受承载上限约束
4. 当前不施加 `L1-C2` 硬约束
5. 超时不被显式禁止，而是通过目标函数中的归一化时延项进行软惩罚
6. 当前不存在 `drop` 失败变量，所有活跃任务都必须被分配

## 7. 最终结论

当前第一层的代码实现已经完成一次语义切换：

- 从“硬约束 + drop 失败机制”
- 切换为“统一分配 + 软惩罚目标”

当前项目的主要问题不是第一层代码本身难以理解，而是文档系统中同时保留了切换前后的两套表述，且部分索引和审查报告没有及时标注“版本上下文”，导致阅读者容易误判当前实现。
