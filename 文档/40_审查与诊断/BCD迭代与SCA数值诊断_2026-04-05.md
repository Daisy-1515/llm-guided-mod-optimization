# BCD 迭代收敛与 SCA 数值稳定性诊断

> 日期：2026-04-05
> 触发：D1 实验中 BCD 仅迭代 2 次即收敛；方案 C 参数下 SCA 求解器崩溃

---

## 1. 问题 A：BCD 只迭代 2 次

### 现象

2026-04-05 所有 D1 实验（15 个 run）均在 k=1 触发收敛退出，cost 完全或几乎不变：

| 类型 | cost_history | relative gap |
|------|-------------|-------------|
| 多数 | `[0.690, 0.690]` | 0.0（完全相同） |
| 少数 | `[13539.26, 13536.90]` | 0.017% < eps_bcd(0.1%) |

收敛判定逻辑（`bcd_loop.py:686-693`）：k=0 跳过检查，k=1 一检查就满足 `relative_gap < eps_bcd`，立刻 `break`。

### 根因：通信时延占比过低，L1 二值决策对轨迹不敏感

**卸载时延构成**（当前参数：B=40MHz, D_l=200kbit, f_edge=1e9 Hz）：

| UAV-任务距离 | 上行时延 | 计算时延 | 下行时延 | 通信占比 |
|---|---|---|---|---|
| 50m | 2.15ms | **150.0ms** | 0.39ms | **1.7%** |
| 100m | 2.77ms | **150.0ms** | 0.48ms | **2.1%** |
| 200m | 5.00ms | **150.0ms** | 0.79ms | **3.7%** |
| 300m | 8.55ms | **150.0ms** | 1.25ms | **6.1%** |

**计算时延 `F/f_edge` 占 94%~98%，完全不依赖轨迹。**

轨迹变化 20m 对卸载时延的影响：

| 任务量级 | D_offload(100m) | D_offload(80m) | 变化量 |
|---|---|---|---|
| 轻 (F=5e7) | 53.25ms | 52.92ms | 0.33ms (0.62%) |
| 中 (F=1.5e8) | 153.25ms | 152.92ms | 0.33ms (0.22%) |
| 重 (F=2.5e8) | 253.25ms | 252.92ms | 0.33ms (0.13%) |

### 结论

L1 MILP 的二值卸载决策由 `F/f_local vs tau` 决定（轨迹无关）。SCA 优化后轨迹变化 10~30m，对 D_hat_offload 影响不到 1%，不足以翻转任何 0/1 决策。BCD 在第一次 pass 后即到达不动点。

## 2. 调参方案

为增强 L1-L2 耦合，需增大通信时延在卸载总时延中的占比。

### 方案 C（推荐）：B=5MHz + 大数据量

```
B_up = 5e6        # 40MHz → 5MHz
B_down = 5e6
D_l_min = 5e5     # 100kbit → 500kbit
D_l_max = 1.5e6   # 300kbit → 1.5Mbit
D_r_min = 1e5     # 20kbit → 100kbit
D_r_max = 5e5     # 80kbit → 500kbit
```

预期效果（中等任务 F=1.5e8）：

| 距离 | 上行时延 | 计算时延 | 下行时延 | 通信占比 | 总时延 |
|---|---|---|---|---|---|
| 50m | 86.1ms | 150.0ms | 12.6ms | 39.7% | 248.8ms |
| 100m | 110.7ms | 150.0ms | 15.5ms | 45.7% | 276.1ms |
| 200m | 200.0ms | 150.0ms | 25.2ms | 60.0% | 375.2ms |
| 300m | 341.9ms | 150.0ms | 40.0ms | 71.8% | **531.9ms > tau!** |

300m 处 D_offload 跨过 tau（500ms），轨迹直接决定卸载可行性 → L1-L2 强耦合。

### 其他方案对比

| 方案 | 通信占比@100m | 时延范围 | 评价 |
|---|---|---|---|
| A. 当前 | 2.1% | 7.2ms | 轨迹无影响力 |

| D. B=10MHz+中数据 | 17.4% | 71ms | 安全保守 |
| E. B=5MHz+降 f_max | 9.5% | 142ms | 全距离超 tau，不推荐 |

## 3. 问题 B：方案 C 下 SCA 求解器崩溃

### 现象

切换到方案 C 参数后运行 D1 实验：

```
SCA iteration 0 failed: all candidate solvers were infeasible or unsafe.
Last status=optimal_inaccurate
```

L1 求解正常（obj=24.97），SCA 轨迹优化在第 0 次迭代即失败。

### 根因：SCA 代理系数爆炸 320x，SOCP 数值病态

`trajectory_opt.py:865-867` 中 SCA 通信代理目标的关键系数为 `D_l / r_up_ref²`：

| | 旧参数 | 新参数 | 放大倍数 |
|---|---|---|---|
| r_up_ref | 7.23e7 bps | 9.04e6 bps | 8x 缩小 |
| **D_l / r_up_ref²** | 3.82e-11 | 1.22e-8 | **320x** |
| **D_r / r_down_ref²** | 7.65e-12 | 3.67e-9 | **480x** |
| ×alpha=35 总放大 | — | — | **~12000x** |

### 失败链条

```
B_up 缩小 8x + D_l 增大 5x
  → r_up_ref 缩小 8x
    → D_l/r_up_ref² 放大 320x
      → SOCP 条件数 κ ~ 1e13（远超 CLARABEL/ECOS 容限 ~1e5）
        → solver 返回 optimal_inaccurate
          → 提取轨迹有投影误差 → 速度约束验证失败
            → 所有候选被拒 → SCA 失败
```

### 代码位置

| 组件 | 文件 | 行号 | 问题 |
|---|---|---|---|
| 代理系数公式 | `trajectory_opt.py` | 854-867 | D_l/r_up_ref² 系数爆炸 |
| 求解器接��逻辑 | `trajectory_opt.py` | 184-257 | 拒绝 inaccurate + 速度不合格的解 |
| 速度验证 | `trajectory_opt.py` | 459-506 | 拦截病态解 |

## 4. 下一步

### 4.1 修复 SCA 系数归一化（优先）

在 `_build_sca_subproblem()` 中对通信代理系数做参考值归一化：

```python
# 计算参考值
D_ref = median(D_l for all active tasks)
r_ref = median(r_up_ref for all active offloads)

# 归一化系数
coeff = (D_l_i / D_ref) / (r_up_ref / r_ref) ** 2
# 保持 O(1) 量级，不随参数配置爆炸
```

### 4.2 放宽 tau（可选配合）

方案 C 下 300m 处 D_offload > tau，可适当放宽：

```
tau_min = 0.6    # 0.5 → 0.6
tau_max = 0.8    # 0.5 → 0.8
```

### 4.3 验证 BCD 迭代次数

修复后重新跑 D1 实验，观察：
- BCD 是否迭代超过 2 次
- L1 卸载决策是否在迭代间发生变化
- cost_history 是否呈现有意义的下降趋势
