# 2026年3月9日 周一

## 项目：边缘无人机计算卸载系统建模

---

今天是这个项目正式动工的第一天。下午两点半坐到电脑前，一直干到晚上七点，算是把数学模型的骨架搭起来了。

### 下午：搭环境、理脉络

最开始的工作是把开发环境配好——Claude、Gemini、Codex 三个 AI Agent 的协作规则，VS Code 的配置，`.gitignore`，这些琐碎但必要的东西。弄完这些差不多三点。

然后开始梳理原论文的项目结构。这个项目脱胎于一篇 NeurIPS 2025 的工作，原本做的是网约车调度优化，用 LLM 引导目标函数演化。我要做的是把场景从"出租车接乘客"换成"无人机执行边缘计算卸载"——核心框架不变，但底层的物理模型完全不同。

花了些时间把项目结构分析写清楚，中英文各一份，又整理了改造计划。这一步看似慢，但后面推公式的时候确实省了不少回头翻找的功夫。

### 傍晚：公式推导与成本建模

三点半开始进入正题——数学建模。

先补上了远程卸载时延的公式，这块原文档里是缺失的。然后给所有公式段落加上了符号解释，不然过几天自己都看不懂。

四点左右开始写第 7 节——系统成本模型。这是整个优化问题的目标函数，把通信时延、计算时延、飞行能耗这些东西统一到一个加权成本里。写的过程中发现飞行能耗需要单独分离出来建模，于是做了一次重构，把公式修正了。流程图也同步更新。

后面又加了优化变量的定义、凸性分析、辐射分析这些内容。到这一步，模型的完整度已经可以了。

### 晚上：做减法

做到六点半的时候，回头审视整个模型，觉得太重了。

第一刀：**砍掉地面基站**。原来的模型里有地面基站作为中继节点，但仔细想想，在我的场景里无人机直接和边缘服务器通信就够了，多一层基站反而增加了不必要的复杂度。去掉之后网络拓扑清爽很多。

第二刀：**去掉终端能耗，固定发射功率为常数**。终端设备的能耗建模会引入一堆新的变量和约束，但对我关注的核心问题——卸载决策和无人机轨迹——贡献有限。发射功率固定之后，信道模型简化了，优化变量少了，问题更容易处理。

最后把公式格式优化了一遍，同步更新了变量映射和清单文档，确保各处一致。

### 小结

今天的核心收获：从零搭建了一个边缘无人机系统的数学模型，经历了"从无到有"再到"做减法"的完整过程。一开始总想把模型做得面面俱到，后来发现简洁的模型才是好模型——该有的物理意义不能丢，但不影响核心决策的东西要果断砍掉。
