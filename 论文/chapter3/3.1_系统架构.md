# 3.1 无人机辅助边缘计算系统架构

在第二章的文献综述中，本文系统梳理了无人机辅助移动边缘计算领域的研究现状，指出现有工作在双向数据传输建模方面的不足。本章将建立完整的系统数学模型。在无人机辅助边缘计算系统中，终端设备将原始数据通过上行链路卸载至无人机，无人机完成边缘计算后再通过下行链路将结果回传至终端，由此形成"上行传输—边缘计算—下行回传"的三段串行流水线。由于无人机在任务处理的各阶段持续移动，导致通信信道状态随时空变化，使得卸载决策、资源分配与轨迹规划三者深度耦合。本节将从系统场景、双向数据流模型、符号约定和基本假设四个方面对系统架构进行完整描述。

## 3.1.1 系统场景描述

本文考虑如图 3-1 所示的无人机辅助移动边缘计算场景。在一个地理范围为 $[0, x^{\max}] \times [0, y^{\max}]$ 的二维矩形服务区域内，$I$ 个地面终端设备（Terminal Device, TD）随机分布于固定位置 $\mathbf{p}_i = [x_i, y_i]^T,\ i \in \mathcal{I} = \{1, 2, \dots, I\}$。这些终端设备受限于电池容量和处理器性能，难以在本地高效处理计算密集型应用（如目标识别、环境三维建模或高清视频分析等）。为此，系统部署 $J$ 架旋翼无人机作为空中移动边缘服务器，定义无人机集合为 $\mathcal{J} = \{1, 2, \dots, J\}$。每架无人机均搭载微型边缘计算服务器，可直接为地面终端提供计算卸载服务；本文不引入独立的地面边缘服务器，即边缘节点集合与无人机集合重合 $\mathcal{J} = \mathcal{U}$[^1]。

[^1]: 本文中 $\mathcal{J}$ 与 $\mathcal{U}$ 可互换使用，$\mathcal{U}$ 强调无人机的物理实体角色，$\mathcal{J}$ 强调其作为边缘计算节点的功能角色。

所有无人机在预设的固定高度 $H$ 进行二维水平飞行，无人机 $j$ 在时隙 $t$ 的水平位置记为 $\mathbf{q}_j^t = [x_j^t, y_j^t]^T$。系统采用离散时间模型，将整个调度周期等分为 $T$ 个时隙，定义时隙集合 $\mathcal{T} = \{1, 2, \dots, T\}$，每个时隙的持续时长为 $\delta$ 秒。在单个时隙内，无人机位移量相对于其与终端的距离而言足够小，因此每个时隙内的无人机位置可近似为恒定。

无人机 $j$ 与终端 $i$ 在时隙 $t$ 的三维欧氏距离为：

$$
d_{i,j}^t = \sqrt{H^2 + \|\mathbf{p}_i - \mathbf{q}_j^t\|^2}
$$

所有无人机从公共基站 $\mathbf{q}^{\mathrm{dep}}$ 出发，完成服务任务后返回同一位置，即满足起止约束 $\mathbf{q}_j^I = \mathbf{q}_j^F = \mathbf{q}^{\mathrm{dep}},\ \forall j \in \mathcal{U}$。每个终端设备并非全程在线，而是在一段连续的活跃窗口内产生计算请求，以活跃标志 $\zeta_i^t \in \{0, 1\}$ 表示终端 $i$ 在时隙 $t$ 是否产生计算任务。

由此可见，本系统架构中存在一个核心权衡：将任务卸载至无人机边缘服务器可以显著降低终端的计算延迟，但同时消耗了有限的无线通信资源和无人机飞行能量。如何在延迟、能耗与轨迹之间实现全局最优平衡，构成了后续建模与求解的核心问题。

## 3.1.2 双向数据流模型

传统的移动边缘计算研究往往假设计算结果的数据量极小而忽略下行回传阶段的延迟。然而，在交互式增强现实、控制反馈和协同感知等实际应用中，下行数据传输对系统整体时延具有不可忽视的影响。为此，本文建立完整的双向数据流模型，将下行回传纳入端到端延迟分析。

对于终端 $i \in \mathcal{I}$ 在时隙 $t \in \mathcal{T}$ 产生的计算任务，本文采用四元组对其特征进行刻画：

$$
U_i^t \triangleq (D_i^l,\, D_i^r,\, F_i,\, \tau_i), \quad i \in \mathcal{I},\ t \in \mathcal{T} \tag{3-1}
$$

其中，$D_i^l$ 为上行卸载数据量（bits），表示终端需上传至无人机的原始数据（如传感器数据流、高清图像帧等）；$D_i^r$ 为下行回传数据量（bits），表示无人机完成计算后需返回终端的处理结果（如控制信令、渲染后图像等）；$F_i$ 为该任务所需的总计算量（CPU cycles）；$\tau_i$ 为任务的最大容忍延迟（s），即从数据开始上传到终端接收最终结果的端到端时限。

当任务 $U_i^t$ 卸载至无人机 $j$ 执行时，其处理过程构成严格的三段串行流水线：

1. **上行传输阶段**：终端 $i$ 将原始数据 $D_i^l$ 通过上行链路传输至无人机 $j$，所需时间为 $T_{i,j}^{\mathrm{up},t}$。
2. **边缘计算阶段**：无人机 $j$ 利用机载计算资源处理 $F_i$ 个 CPU 周期，所需时间为 $T_{i,j}^{\mathrm{comp},t}$。
3. **下行回传阶段**：无人机 $j$ 将计算结果 $D_i^r$ 通过下行链路回传至终端 $i$，所需时间为 $T_{i,j}^{\mathrm{down},t}$。

由于数据处理天然具有信息浓缩的特征，原始数据量通常远大于处理结果的数据量，即 $D_i^l \gg D_i^r$，这使得上行传输阶段成为通信时延的主要瓶颈。更重要的是，由于三个阶段在时间上串行展开，无人机在各阶段的位置发生变化：上行传输时无人机位于 $\mathbf{q}_j(t_1)$，计算完成时移动至 $\mathbf{q}_j(t_2)$，下行回传时到达 $\mathbf{q}_j(t_3)$。空地距离的动态变化导致信道增益随之改变，从而使上行速率与下行速率呈现时空不对称性。这种由任务流水线的串行结构与无人机物理运动共同引发的**时空耦合效应**，是本系统"双向数据计算"特性的核心体现，也是后续建模需要重点刻画的关键机制。

## 3.1.4 基本假设

为使所建立的数学模型既能反映物理系统的核心特征，又具备计算上的可处理性，本节对系统运行环境提出以下七条基本假设：

**A1（视距信道主导）**：由于无人机在高度 $H \ge 100$ m 处飞行，空地链路以视距（Line-of-Sight, LoS）传播为主，可忽略严重的多径衰落和非视距阻塞。信道增益主要由自由空间路径损耗决定，与距离的平方成反比。该假设在城郊和空旷地区的中高空无人机通信场景中被广泛采用。

**A2（固定高度飞行）**：所有无人机均在同一预设高度 $H$ 进行二维水平面飞行。该假设将三维轨迹优化降维为二维问题，大幅降低了求解复杂度，同时避免了频繁爬升和俯冲带来的额外能耗与安全隐患。

**A3（OFDMA 正交接入）**：系统在空口采用正交频分多址（Orthogonal Frequency Division Multiple Access, OFDMA）技术，为各条上行/下行通信链路分配正交频谱子带，因此不考虑同频干扰。该假设使得通信速率仅取决于信道质量和分配带宽，简化了干扰管理建模。

**A4（准静态信道）**：在单个时隙 $\delta$ 内，无人机位置可近似视为恒定。由此，每个时隙内的信道增益和传输速率为常数，信道状态仅在时隙切换时更新。该假设在时隙 $\delta$ 远小于信道相干时间时成立。

**A5（单时隙卸载决策）**：每个活跃时隙内独立做卸载决策，即任务在产生的当前时隙内完成上行传输、边缘计算和下行回传的完整流程，三段串行延迟之和不超过任务截止期 $D_{i,j}^t \le \tau_i$。该假设避免了复杂的跨时隙任务状态迁移和缓存队列建模，使各时隙的资源分配问题可解耦求解。

**A6（公共基站起止约束）**：所有无人机从同一公共基站 $\mathbf{q}^{\mathrm{dep}}$ 出发，完成服务周期后返回同一位置，即 $\mathbf{q}_j^1 = \mathbf{q}_j^T = \mathbf{q}^{\mathrm{dep}},\ \forall j \in \mathcal{U}$。该基站同时承担无人机充电补给和全局调度信令下发的角色。

**A7（连续活跃窗口）**：每个终端设备的任务产生期为一段连续的时隙窗口。在活跃窗口 $[\zeta_i^t = 1]$ 内，终端每个时隙产生一个计算任务；窗口外终端处于静默状态，不产生卸载请求。该假设符合实际中终端执行连续数据采集任务（如视频录制、环境扫描）的业务特征。
